# CellRebelAuto MVP 复盘

## 复盘范围

本次复盘覆盖当前仓库中可见的 MVP 开发阶段，主要依据以下事实源：

- `git log`
- tag `v0.1.0-mvp`
- `HANDOFF.md`
- `CHANGELOG.md`
- `docs/PROJECT_ANALYSIS.md`
- `docs/ROADMAP.md`
- 当前 Android 源码结构与关键实现

本复盘的目标不是追责，而是回答五个问题：

1. 我们原本想做什么
2. 实际做成了什么
3. 哪些地方做得好
4. 哪些地方偏了、为什么偏
5. 下一阶段最值得优先修正什么

## 一句话结论

这个 MVP 的成功之处在于，我们已经把一个真实可运行的 Android 自动化工具从 0 搭出来了；它的主要失败点不在功能缺失，而在低估了目标设备环境，尤其是 MIUI 对跨 App 自动化的破坏性。

## 1. 目标与结果对照

### 原始目标

从代码结构、tag 注释和 handoff 反推，这个项目的 MVP 目标是：

- 在 Android 设备上通过 AccessibilityService 协调 Fake GPS 与 CellRebel
- 在一个设定好的 GPS 范围内循环执行测试
- 采集每轮测试结果并本地保存
- 提供基本操作界面，方便启动、停止、查看结果和导出

### 实际结果

当前仓库已经完成的部分：

- Android App 骨架已建立，有可运行的 Compose UI
- 已实现配置、控制面板、历史记录、CSV 导出、日志导出、A11y 树导出
- 已实现自动化主循环、Fake GPS 处理器、CellRebel 处理器、Room 数据落盘
- 已有一个明确的 MVP tag：`v0.1.0-mvp`

当前没有真正完成收口的部分：

- 跨 App 切换在目标 MIUI 环境上仍不稳定
- 端到端“连续多轮稳定运行”的可信度不足
- 版本口径、发布口径和正式文档口径还没有统一
- 缺少自动化测试，回归成本偏高

### 结果判断

如果按“有没有做出一个可用原型”衡量，这次 MVP 是成功的。
如果按“能不能稳定地产出可信测试数据”衡量，这次 MVP 还是部分完成。

这两个判断必须同时成立，单独强调任意一边都会失真。

## 2. 这次做对了什么

### 2.1 先把主链路搭通，而不是过早做外围优化

项目没有陷入“先写完整架构、再碰设备问题”的路径，而是很快把主链路拼出来了。这个选择是对的，因为真正的难点最终不在本地 UI、数据库或导出功能，而在设备行为。

### 2.2 调试能力建设是正确的 scope creep

日志导出和无障碍树导出不是最初的核心功能，但对这个项目来说非常值。
因为它面对的是两个第三方 App 和一个 OEM 系统层行为，不可观测性本身就是成本。补这些能力属于 justified 的 scope creep，而不是分心。

### 2.3 架构层次已经有雏形

当前结构已经自然分成：

- UI 层
- 服务层
- 自动化编排层
- 第三方 App 适配层
- 底层桥接层
- 本地存储层

这说明即使是在 MVP 阶段，代码也没有完全坍缩成一团设备脚本，后续仍具备继续演化的基础。

### 2.4 很快识别出“系统环境是主风险”

从 `HANDOFF.md` 看，团队已经比较快意识到：

- 问题不是 CellRebel 本身不能点
- 也不是 Fake GPS 完全不能用
- 而是 MIUI 对切换与启动行为做了额外约束

这个识别本身很关键，因为它把问题从“继续乱试 UI 操作”转成了“理解系统限制并设计绕行方案”。

## 3. 这次没做好的地方

### 3.1 低估了 OEM 环境对需求复杂度的影响

这是这次项目最核心的偏差。

从功能描述上看，这个工具像一个“简单的跨 App 自动化器”；但一旦落在 MIUI / HyperOS 上，复杂度会被系统策略放大。我们一开始更像是在按标准 Android 假设推进，后面才发现目标环境不是“普通 Android”，而是“带强约束的 OEM Android”。

### 3.2 成功标准定义得还不够硬

当前阶段很容易把“流程跑通一次”理解为成功，但对这个项目来说，一次成功不够。

真正应该前置的里程碑是：

- 在目标设备上连续跑多轮
- 能导出结构完整的数据
- 失败率在可接受范围内

如果没有这个更硬的标准，团队会自然高估 MVP 完成度。

### 3.3 文档和版本口径滞后于代码推进

项目已经有：

- git tag
- `CHANGELOG.md`
- `docs/ROADMAP.md`

但这些是在后补的。开发进行时，发布口径、版本口径和下一阶段边界没有同步收敛，结果是：

- `versionName = "1.0"` 与 `v0.1.0-mvp` 不一致
- “是否已公开发布”这个问题在文档上需要靠假设收口
- 原始 MVP 范围需要事后重建，而不是顺手可读

### 3.4 测试保护为空

当前仓库没有自动化测试，这意味着：

- 逻辑修改容易打回设备验证
- 交互回归难以及时发现
- 每次修 Bug 的反馈回路偏慢

对普通 App 来说，这已经是风险；对一个高度依赖设备行为的自动化工具来说，风险更大。

## 4. 根因分析

### 根因一：把“功能复杂度”看得比“环境复杂度”更重

本项目的主难点不是功能设计，而是运行环境。
如果一开始就把 MIUI 视作一等约束，很多验证顺序会不同：

- 会更早验证跨 App 切换
- 会更早确认最近任务、权限、前后台策略
- 会更早决定是否需要 shell / ConfirmStartActivity 兜底

### 根因二：缺少一个更强的阶段性验收标准

当前 MVP 的“完成”更像是：

- 功能模块都已经有了

但真正应该采用的验收标准是：

- 目标设备上能连续稳定执行
- 有导出结果可作为证据
- 已知失败模式可复现、可解释、可绕过

如果没有后者，项目就会在“结构上很完整”和“结果上还不稳定”之间产生认知落差。

### 根因三：项目知识沉淀略晚

很多关键认知确实被记录下来了，但它们更多是后置沉淀，而不是前置约束。

例如：

- 版本口径
- MVP 目标边界
- 下一阶段明确不做项
- 目标设备环境的特殊性

这些如果更早进入主文档，会减少后面重建叙事的成本。

## 5. 经验教训

### 5.1 对设备自动化项目，先验证最危险的环境假设

以后类似项目，最先做的不是把所有页面做出来，而是先验证：

- 能不能切换目标 App
- OEM 是否拦截关键行为
- 权限模型是否可控
- 失败后是否可恢复

先把“环境可行性”打穿，再扩展业务功能。

### 5.2 调试能力应该被当成主功能的一部分

对这类项目来说，可观测性不是锦上添花，而是开发能力本身。

日志、A11y dump、导出结果、失败路径诊断，应该继续保留，甚至前置增强。

### 5.3 成功标准必须可证据化

下一阶段再说“完成”，至少要带着这些证据之一：

- 连续成功运行次数
- 导出的 CSV 样本
- 明确的失败率
- 指向具体设备/ROM 版本的复现实验记录

### 5.4 文档口径要跟版本推进同步

tag、`versionName`、release 说明、roadmap 目标，这四者不能继续分离。
否则每次复盘都要重新解释“这个版本到底算什么”。

## 6. 下一阶段最重要的行动项

### P0

1. 把“目标设备上连续 20 次成功跑通”定义为下一阶段的硬门槛
2. 系统化验证 MIUI 权限、前后台限制、Recent Apps 路径和 ConfirmStartActivity 处理方案
3. 产出一份带 CSV 证据的完整端到端成功样本

### P1

4. 统一版本口径，决定未来到底使用 `1.0` 还是 `0.1.0`
5. 给非 UI 核心逻辑补上最小测试保护
6. 把运行配置持久化，减少每次设备重启后的人工重复输入

### P2

7. 明确下一阶段暂时不支持哪些设备、系统和分发方式
8. 视稳定性结果决定是否值得推进 Play Store / 更广设备支持 / 远程指标能力

## 7. 最终判断

这次 MVP 不属于“失败的原型”，也不属于“已经完成的产品化版本”。

更准确的定义是：

- **它已经证明这个方向在工程上是可做的**
- **但还没有证明它在目标环境上是稳定可依赖的**

因此，下一阶段不应该急着加更多功能，而应该优先把“可重复、可证明、可维护”补齐。

如果这一点补上，这个项目才会从“能演示的工具”真正进入“能持续使用的工具”。
